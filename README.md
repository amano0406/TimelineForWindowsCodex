# TimelineForWindowsCodex

## Current Role

`TimelineForWindowsCodex` converts local Windows Codex Desktop history into per-thread Timeline-ready JSON artifacts. It preserves the user / assistant / system message chain so another LLM or the downstream Timeline product can consume it.

The CLI is still the primary product surface. API migration has been rolled back except for one minimal C# health endpoint:

```text
GET http://localhost:<runtime.apiPort>/health
```

The health endpoint returns only a JSON boolean: `true` or `false`.

## Scope

The source Codex history is read-only. This product writes only to:

- the configured `outputRoot`
- optional download ZIP folders
- local Docker/app runtime data

`items remove` deletes generated master item directories only. It does not delete source Codex history or existing ZIP exports. Removed items can be regenerated by running `items refresh`.

## Input

The worker reads fixed Codex history roots through Docker Compose read-only mounts.

- Current Codex home: `C:\Users\amano\.codex`
- Known archived Codex home: `C:\Codex\archive\migration-backup-2026-03-27\codex-home`

The main transcript source is `sessions/**/*.jsonl`. `session_index.jsonl`, archived `thread_reads`, and `state_5.sqlite` fallback metadata are used only where appropriate for discovery and metadata.

## Output

The primary output is the master artifact directory configured by `settings.json`.

Current local configuration points to:

```text
C:\apps\Timeline\data\to_text\windows-codex
```

Master output layout:

```text
<outputRoot>/
  <thread_id>/
    convert_info.json
    timeline.json
```

Download ZIP layout:

```text
README.md
items/
  <thread_id>/
    convert_info.json
    timeline.json
```

See [Outputs](docs/OUTPUTS.md) for the exact `timeline.json`, `convert_info.json`, and CLI JSON response structures.

## Settings

`settings.json` uses this top-level order:

```json
{
  "schemaVersion": 1,
  "runtime": {
    "instanceName": "",
    "apiPort": 19200
  },
  "outputRoot": "C:\\apps\\Timeline\\data\\to_text\\windows-codex"
}
```

- `runtime.instanceName` controls the Docker Compose project name suffix.
- `runtime.apiPort` controls the local health API port.
- `outputRoot` controls where generated master item directories are written.
- Unknown fields are preserved when settings are updated.

## Quick Start

Run from the repository root:

```powershell
cd C:\apps\TimelineForWindowsCodex
.\start.bat
.\cli.bat settings status
.\cli.bat items refresh --json
.\cli.bat items download --to C:\apps\Timeline\data\downloads\windows-codex --json
```

Health check:

```powershell
Invoke-RestMethod http://localhost:19200/health
```

Stop the services:

```powershell
.\stop.bat
```

## Common Commands

Use the `.bat` launchers on Windows:

```powershell
.\cli.bat settings init
.\cli.bat settings status
.\cli.bat settings master show
.\cli.bat settings master set C:\apps\Timeline\data\to_text\windows-codex

.\cli.bat items list --json
.\cli.bat items list --page 2 --page-size 50 --json
.\cli.bat items refresh --json
.\cli.bat items refresh --download-to C:\apps\Timeline\data\downloads\windows-codex --json
.\cli.bat items download --to C:\apps\Timeline\data\downloads\windows-codex --json
.\cli.bat items remove --item-id <thread_id> --json
```

## Testing

Run the main operational check suite:

```powershell
.\test-operational.bat
```

The operational suite uses fixture Codex homes under:

- `tests/fixtures/codex-home-min`
- `tests/fixtures/archived-root-min`

It covers CLI refresh/download, source-to-output fidelity, Windows launcher flow, Docker Compose refresh reuse, and `items remove` against temporary fixture output.

Additional direct checks:

```powershell
dotnet build api\TimelineForWindowsCodex.HealthApi\TimelineForWindowsCodex.HealthApi.csproj
python -m unittest discover -s worker\tests -v
docker compose config --quiet
```

## Known Current Limits

- Full API migration is intentionally not active. Only `/health` exists.
- `cli.ps1` / `cli.bat` are still required for normal operation.
- `items list` can be slow on large real Codex history because discovery still scans source history before pagination is applied.
- Binary attachment contents are not exported.
- Tool-call details, terminal output, and reasoning summaries are not exported into `timeline.json`.
